# Spec: Q10

## Objetivo funcional

Integrar CIUNAC Backend con Q10 para crear estudiantes y sincronizar horarios/cursos.

## Actores

- Administrador academico.
- Sistema Q10.
- Frontend administrativo.

## Precondiciones

- `API_KEY` valida para consumir endpoints del backend.
- `API_KEY_Q10` configurada para consumir Q10.
- Datos minimos del estudiante disponibles.

## Flujo principal

1. El frontend solicita crear estudiante en `POST /q10/estudiantes`.
2. `Q10Service` invoca Q10 con credenciales configuradas.
3. Si Q10 crea el estudiante, se registra en programa cuando aplique.
4. El administrador consulta horarios con `GET /q10/horarios-cursos`.
5. El administrador guarda horarios/cursos con `POST /q10/horarios-cursos`.

## Flujos alternos y errores

| Caso | Resultado |
| --- | --- |
| Q10 rechaza creacion | `400 Bad Request` con informacion controlada. |
| Falla al listar horarios | `400 Bad Request`. |
| Falla al guardar horarios | `400 Bad Request`. |
| API key backend invalida | `401 Unauthorized`. |

## Reglas de negocio

- La integracion Q10 queda encapsulada en `Q10Service`.
- Errores de Q10 deben transformarse a excepciones controladas.
- No deben registrarse secrets de Q10 en logs.

## Contratos API

| Metodo | Endpoint | Auth | Request | Response |
| --- | --- | --- | --- | --- |
| `POST` | `/q10/estudiantes` | ApiKey | `Q10EstudianteDto` | Resultado de creacion/registro |
| `GET` | `/q10/horarios-cursos` | ApiKey | Query opcional | Horarios/cursos |
| `POST` | `/q10/horarios-cursos` | ApiKey | `Q10HorariosDto` | Resultado de guardado |

## Datos involucrados

- DTOs: `q10-estudiante.dto.ts`, `q10-horarios.dto.ts`.
- Entidades relacionadas segun guardado de horarios/cursos.

## Integraciones externas

- Q10 API externa.

## Criterios de aceptacion

- Dado un estudiante valido, cuando se envia a Q10, entonces se devuelve resultado de creacion o error controlado.
- Dado una falla Q10, cuando ocurre, entonces el backend responde `400` sin exponer secrets.
- Dado horarios validos, cuando se guardan, entonces quedan disponibles para operacion interna.

## Casos de prueba sugeridos

- Unit: mock de llamadas Q10 exitosas y fallidas.
- Integration: endpoint con API key y payload valido.
- E2E: guardar horarios y consultar respuesta esperada.
